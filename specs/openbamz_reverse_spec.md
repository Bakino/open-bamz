## 1. Technology Stack and Architecture

### 1.1 What Open BamZ is

Open BamZ is **both an application runtime and a plugin manager** (`doc/README.md:5`). End
users use it to create *applications*; each application is a self-contained unit composed
of:

- a **dedicated PostgreSQL database**, and
- a **file directory** holding the application's **front-end source code** (`doc/README.md:24-33`).

The core platform does only a small, well-defined set of things; everything else is
delivered through plugins (`doc/README.md:223-230`):

1. Manage user accounts.
2. Create/delete application databases and directories.
3. Expose a GraphQL API per application (via PostGraphile).
4. Serve application files and inject the BamZ runtime + plugins into them.

### 1.4 The two-database model (observed)

**Main database** (`process.env.DB_NAME`, default `_openbamz`):
- `private.account` — user accounts (`src/database/_openbamz.sql:26-34`).
- `private.session` — refresh-token sessions (`src/database/_openbamz.sql:75-82`).
- `public.app` — the application registry: `code`, `name`, `owner`, `hosts`, `admins` (`src/database/_openbamz.sql:245-254`).

**Application database** (one per app, named after `app.code`):
- `openbamz.plugins` — list of activated plugins + version (`src/database/init_base.sql:62-66`).
- `public.*` — application tables created by the developer.
- `<plugin>` schemas — created by plugins for their own data.
- `graphile_worker.*` — per-database job queue.

### 1.5 Event-driven control plane (observed)

State changes in the database are propagated to the Node.js layer **through
`graphile_worker` jobs enqueued by `plv8` triggers**. This is the backbone of the system:

| DB event (trigger) | Enqueues job | Worker task | Effect |
|--------------------|--------------|-------------|--------|
| `INSERT public.app` | `createDatabase` | `tasks/createDatabase.js` | create DB, schema, role, privileges, directory, start workers |
| `DELETE public.app` | `dropDatabase` | `tasks/dropDatabase.js` | drop DB + roles + directory, clear caches |
| `INSERT/UPDATE public.app` | `updateHostnameCache` | `tasks/updateHostnameCache.js` | refresh in-memory hostname→app map |
| `INSERT openbamz.plugins` | `addPlugin` | `tasks/addPlugin.js` | run plugin `prepareDatabase`, reset GraphQL |
| `DELETE openbamz.plugins` | `removePlugin` | `tasks/removePlugin.js` | run plugin `cleanDatabase`, reset GraphQL |
| plugin-defined | `runPluginTask` | `tasks/runPluginTask.js` | run an arbitrary plugin task file |

Evidence: triggers in `src/database/_openbamz.sql:300-418` and
`src/database/init_base.sql:70-93`; tasks in `src/database/tasks/`.

## 3. Observed Requirements (EARS Format)

### 3.1 Bootstrap and main database

- The system shall read all database connection parameters from environment variables
  (`DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`, `DB_NAME`). — `src/index.js:31-37`
- When `MONO_DATABASE` is set, the system shall skip main-database preparation and start
  workers only for that single database. — `src/index.js:41-50`
- When started normally, the system shall create the main database if absent, create the
  base roles, apply `_openbamz.sql`, grant main roles, and start all workers. —
  `src/index.js:52-63`
- When the main database does not exist, the system shall connect to the `postgres`
  database and issue `CREATE DATABASE`. — `src/database/init.js:24-37`
- The system shall start a `graphile-worker` runner per application database with
  concurrency 5 and a 1000 ms poll interval. — `src/database/init.js:529-551`

### 3.2 Application-name resolution

- When a request arrives, the system shall resolve the target application name in this
  order: hostname cache → `public.app.hosts` lookup → `app-name` header → `?appName=`
  query param → default `DB_NAME`. — `src/index.js:229-291`
- Where `MONO_DATABASE` is set, the system shall force `req.appName` to that database. —
  `src/index.js:230-235`
- When an explicit `appName` references an app that does not exist, the system shall
  respond `Application <name> not found`. — `src/index.js:270-279`
- The system shall set an `app-name` response header equal to the resolved application
  name. — `src/index.js:233`, `src/index.js:289`
- The system shall cache hostname→app and app records in memory; the caches are
  invalidated by worker tasks on app create/update/delete. —
  `src/appCache.js`, `src/database/tasks/updateHostnameCache.js`, `src/database/tasks/dropDatabase.js:12-20`

### 3.3 Authentication and authorization

- When `POST /auth/login` receives a valid email/password, the system shall set
  HttpOnly cookies `jwt-bamz-access` (signed RS256 JWT) and `jwt-bamz-refresh`
  (random token persisted in `private.session`). — `src/auth.js:84-93`, `src/auth.js:56-82`
- The access token shall carry `{ role: account._id }`, audience `bamz-api`, issuer
  `bamz`, and a default TTL of 3 hours (`COOKIE_TTL`). — `src/auth.js:56-59`, `src/auth.js:16-23`
- When `POST /auth/refresh` receives a valid, non-revoked, non-expired refresh token,
  the system shall rotate it (revoke old, issue new) and re-issue cookies. — `src/auth.js:96-110`
- When `POST /auth/logout` is called, the system shall revoke the refresh token and clear
  both cookies. — `src/auth.js:113-124`
- For every request, the `jwtMiddleware` shall verify the access cookie and confirm the
  refresh session is still valid before populating `req.jwt.bamz`. — `src/auth.js:132-156`
- While `MONO_DATABASE` is set, the system shall NOT mount `/auth` routes. — `src/index.js:223-226`
- Credential verification shall be performed in-database via `public.authenticate(email,
  password)` using `pgcrypto crypt()`. — `src/auth.js:48-54`, `src/database/_openbamz.sql:99-121`
- When checking app access, the system shall run a main-DB GraphQL query
  `app_by_code(code)` under the user's role and treat a matching `code` as authorized,
  otherwise raise 401. — `src/database/graphql.js:60-89`

### 3.4 Database access mapping (GraphQL roles)

- The system shall set the PostgreSQL `role` to `anonymous` by default for GraphQL
  requests. — `src/database/graphileConfig.js:92`, `src/database/graphileConfig.js:156`
- Where a `bamz` JWT is present, the system shall set `role` to `jwt.bamz.role` (the
  account UUID) and expose JWT claims as `pgSettings` keys `jwt.bamz.<claim>`. —
  `src/database/graphileConfig.js:94-101`, `src/database/graphileConfig.js:160-173`
- Where a non-`bamz` JWT token carries a role and the request did not originate from a
  `/plugin/` referer, the system shall prefer that application-specific role over the
  bamz role. — `src/database/graphileConfig.js:174-181`
- Server-side calls shall be able to force a role via `contextValue.forceRole`. —
  `src/database/graphileConfig.js:104-107`, `src/database/graphql.js:166-171`
- The per-app PostGraphile service shall connect using the superuser connection string
  because an unprivileged DB user cannot `SET ROLE` to admin roles it does not own. —
  `src/database/graphileConfig.js:121-127`

### 3.5 GraphQL endpoint lifecycle

- When a request hits `/graphql/<app>` or `/graphiql/<app>` for an app other than the
  main DB, the system shall lazily initialize that app's PostGraphile server. —
  `src/index.js:473-518`
- Initializing an app's database shall, in order: apply `init_base.sql`, prepare
  privileges, prepare plugins, then build the PostGraphile handler. —
  `src/database/graphql.js:206-228`
- The set of GraphQL-exposed schemas shall always include `public` and `openbamz`, plus
  any schema whose name equals an activated plugin id (prefix `open-bamz-` stripped),
  plus any schema listed in a plugin's `graphqlSchemas`. — `src/database/graphql.js:118-139`
- The main database GraphQL shall be exposed at `/graphql/<DB_NAME>` with watch mode. —
  `src/database/graphileConfig.js:73-78`

### 3.6 File serving and BamZ injection

- When a request targets a `*.html` file or a directory (`/`), the system shall read the
  file and inject the BamZ runtime before responding. — `src/index.js:353-430`, `src/utils.js:13-88`
- The injected runtime shall set `window.BAMZ_APP`, provide `bamzWaitLoaded()` /
  `bamzGetPlugin()`, and load `/_openbamz_admin.js?appName=<app>`. — `src/utils.js:14-35`
- When the served HTML contains Liquid tags (`{% %}`) and a request is present, the
  system shall render it with request headers as context. — `src/utils.js:70-85`
- The system shall serve each plugin's front-end statics under `/plugin/<pluginId>/`. —
  `src/index.js:433-440`
- When an unauthenticated user requests a plugin file, the system shall serve it only if
  the path is declared in the plugin's `frontEndPublic`; otherwise respond 403. —
  `src/index.js:380-399`
- The system shall serve application files from `$DATA_DIR/apps/<app>/public`, and from
  `src/open-bamz-front` for the default app. — `src/index.js:340-345`, `src/index.js:550-583`
- The system shall reject path traversal by verifying the resolved file path stays within
  the app's public directory. — `src/index.js:562-565`
- Where `MONO_DATABASE` is set, the system shall skip HTML injection and plugin static
  registration (exported sources already embed the runtime). —
  `src/index.js:354-357`, `src/index.js:435`

### 3.7 Admin menu injection

- When `/_openbamz_admin.js` is requested, the system shall emit JavaScript that loads
  every activated plugin's `frontEndLib` into `window.BAMZ_PLUGINS`, builds the admin
  menu, sets `window.BAMZ_LOADED`, and dispatches `openbamz.plugin.loaded`. —
  `src/pluginManager.js:334-416`
- The base menu shall always contain Home / Settings / All apps entries. —
  `src/pluginManager.js:311-325`
- While the referer is inside `/plugin/<app>/`, the system shall NOT re-load plugin libs
  (to avoid recursion), unless `?forceLoadPlugins` is set. — `src/pluginManager.js:381-393`
- The admin top bar shall render only when the current user is an admin of the current
  app (verified by a GraphQL `app_by_code` call). — `src/menu-front/adminMenu.js:5-46`

### 3.8 Plugin lifecycle

- On startup, the system shall scan `PLUGINS_DIR`, import each plugin, topologically sort
  plugins by `package.json openbamz.depends`, then call `initPlugin` on each. —
  `src/pluginManager.js:145-201`
- When plugins have circular dependencies, the system shall log a warning and ignore the
  cyclic plugins. — `src/pluginManager.js:480-485`
- A plugin entry module shall be loaded as ESM when its `package.json` has
  `"type":"module"`, otherwise via CommonJS `require`. — `src/pluginManager.js:46-52`
- When a plugin row is inserted, the system shall first insert any declared
  `plugin.dependencies` (ON CONFLICT DO NOTHING), then run `prepareDatabase`. —
  `src/database/init.js:225-256`
- On each startup, `preparePlugins` shall call a plugin's `prepareDatabase` only when the
  stored version differs from the package version. — `src/database/init.js:204-213`
- When a plugin is removed, the system shall call the plugin's `cleanDatabase`. —
  `src/database/init.js:263-275`

### 3.9 Application file system API

- The `appFileSystems.getFileSystem(app)` helper shall read/write/remove files under
  `$DATA_DIR/apps/<app>/<branch>` (default branch `public`), creating directories as
  needed. — `src/appFileSystems.js:11-56`
- When a file is written or removed, the system shall emit `fileWritten` / `fileDeleted`
  events carrying `{appName, filePath, relativePath, branch}`. — `src/appFileSystems.js:32-56`
- Listeners registered on `appFileSystems` shall be propagated to all current and future
  per-app file systems. — `src/appFileSystems.js:96-104`

### 3.10 Realtime

- The system shall initialize a socket.io server on the HTTP server and support
  `joinRoom` / `leaveRoom` messages. — `src/websocket.js:5-18`, `src/index.js:589`
- The `io` instance shall be passed to plugins (`initPlugin`) and to plugin tasks
  (`runPluginTask`). — `src/index.js:348`, `src/database/tasks/runPluginTask.js:33`

### 3.11 CORS

- The system shall allow all origins with credentials and a fixed base set of
  allowed/exposed headers, dynamically extended by requested headers and by each
  activated plugin's `cors` declaration. — `src/index.js:293-334`

## 4. Non-Functional Observations

- **Connection pooling:** `getDbClient` caches one `pg.Pool` per unique connection-option
  JSON string and wraps `client.query` to log failing queries. — `src/database/dbAccess.js:14-47`
- **Security model leans on PostgreSQL RBAC + RLS.** Access control is enforced by
  database roles and row-level policies (`public.app` policies in
  `src/database/_openbamz.sql:259-278`) rather than application-layer checks alone.
- **Passwords:** account login passwords are hashed with `crypt(..., gen_salt('md5'))`;
  each account also gets a generated DB user with a random UUID password. —
  `src/database/_openbamz.sql:37-54`, `src/database/_openbamz.sql:48`
  > ⚠️ `gen_salt('md5')` is a weak password-hashing choice (see §6).
- **Error masking disabled for app GraphQL** to help admins debug — raw errors are
  returned. — `src/database/graphileConfig.js:136-141`
- **Reserved app codes:** `__backups`, `$bamz`, `bamz`, `app`, `backup`, `api`,
  `creator`, `restore`, `run`, `stats`, `metabase`, `openbamz`, `_openbamz`. —
  `src/database/_openbamz.sql:314-317`
- **Logging:** JSON logs in production, colorized in dev; optional rotating file via
  `LOG_PATH` / `ACCESS_LOG_PATH` with 90-day retention; access log masks `token` /
  `password` fields. — `src/logger.js:44-60`, `src/index.js:155-202`
- **Body limit:** JSON and urlencoded bodies capped at 50 MB. — `src/index.js:219-221`
- **Plugin load ordering** uses Kahn's topological sort. — `src/pluginManager.js:432-486`
- **Per-app GraphQL instances are cached** (`pglByDb`, `graphServers`) and released on
  `clearDatabase`. — `src/database/graphql.js:91-99`, `src/database/graphql.js:235-252`

## 5. Inferred Acceptance Criteria

These are reasonable acceptance criteria inferred from the implementation (validate with
maintainers):

1. Creating a row in `public.app` (as an authenticated user) results, within a few
   poll cycles, in a new database, roles, privileges, an `apps/<code>/public/index.html`,
   and a running worker. — inferred from `src/database/_openbamz.sql:300-331` +
   `src/database/tasks/createDatabase.js`.
2. Deleting that row drops the database, the three roles, and the directory, and clears
   caches. — `src/database/tasks/dropDatabase.js`.
3. Inserting a `plugin_id` into `openbamz.plugins` activates the plugin (its
   `prepareDatabase` runs and its schema becomes GraphQL-visible). — `addPlugin` chain.
4. An authenticated app admin sees the BamZ top bar on app HTML pages; an anonymous
   visitor does not. — `src/menu-front/adminMenu.js:42-46`.
5. GraphQL queries run with the correct PostgreSQL role so that RLS/grants apply per
   user. — `src/database/graphileConfig.js`.
6. A plugin's `/plugin/<id>/...` static files are blocked for anonymous users unless
   listed in `frontEndPublic`. — `src/index.js:380-399`.

## 6. Uncertainties and Questions

1. **`appCache` aliasing:** `src/appCache.js` exports `appCache` as the *same object* as
   `hostnameCache` (`module.exports.appCache = hostnameCache`). Code treats them as
   distinct maps (`appCache[appName] = appRecord` vs `hostnameCache[host] = appName`).
   Sharing one object means hostname keys and app-code keys live in the same map. **Is
   this intentional or a latent bug?**
2. **Duplicated trigger name:** in `_openbamz.sql` the trigger `app_update_admins` is
   declared twice (`:355-358` and `:368-371`), the second time bound to
   `app_update_hostname_cache` logic. The second `CREATE OR REPLACE TRIGGER` reuses the
   name `app_update_admins`, which would **overwrite** the first trigger rather than add
   a hostname-cache trigger. Likely a copy/paste bug — needs confirmation.
3. **Weak password hashing:** `gen_salt('md5')` (`_openbamz.sql:48`) is not suitable for
   password storage. Was `bcrypt`/`argon2` intended?
4. **Hardcoded cookie domain default** `.test3.bakino.fr` (`src/auth.js:71,79,118,121`).
   Production deployments must set `COOKIE_DOMAIN`; is there a guard to prevent shipping
   the default?
5. **`MONO_DATABASE` export mode:** several branches assume "exported sources already
   embed bamz lib" (`src/index.js:354`, `:435`). The export/packaging tool that produces
   those sources is **not present in this repository** — presumably a plugin.
6. **`open-bamz-front` vs `openbamz-front`:** which is canonical, and is the prefix
   stripping in `graphql.js:122-124` a permanent convention or migration scaffolding?
7. **`hasPlugin` / `contextOfApp` connection usage:** `contextOfApp` opens a DB client per
   call (`src/pluginManager.js:99`) and is invoked from hot paths (CORS middleware,
   menu JS). Caching exists for `hasCurrentPlugin` but not for the full context — is the
   per-request DB round-trip acceptable under load?
8. **SQL identifier interpolation:** role/schema/database names are interpolated directly
   into SQL strings throughout `init.js` and into `plv8` template literals in
   `_openbamz.sql`. App `code` is sanitized at creation (`_openbamz.sql:287`,
   `:306`), but plugin-provided schema names in `grantDefaultAccess` are not — confirm
   plugin schema names are trusted input.

## 7. Recommendations

1. **Fix the duplicated `app_update_admins` trigger** so the hostname-cache trigger is
   actually created (rename to `app_update_hostname_cache`). Without it,
   `updateHostnameCache` jobs may never be enqueued on app update.
2. **Replace `gen_salt('md5')`** with `gen_salt('bf', <cost>)` (bcrypt) or move hashing
   to a vetted KDF.
3. **Clarify `appCache.js`** — if two distinct caches are intended, export two separate
   objects.
4. **Guard the default cookie domain** — fail fast in production if `COOKIE_DOMAIN` is
   unset rather than defaulting to a hardcoded host.
5. **Add an automated smoke test** exercising the full app-creation event chain
   (insert app → DB exists → file exists → GraphQL responds), since this flow spans SQL
   triggers, workers, and Node and currently has no visible tests.
6. **Document the `MONO_DATABASE` export pipeline** and where the exporting plugin lives.
7. **Cache `contextOfApp` results** with an explicit invalidation hook
   (`clearCache` already exists in `pluginManager.js:18-20`) to remove per-request DB
   round-trips on hot middleware.
8. **Centralize SQL-identifier quoting** (a single `quoteIdent` helper) to make the
   trust boundary for schema/role/database names explicit.
